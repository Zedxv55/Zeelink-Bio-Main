1. ภาพรวม (OVERVIEW)
--------------------------------------------------------------------------------
Zeelink คือ "แพลตฟอร์มพอร์ตโฟลิโอเชิงสังคมสำหรับคนไทย" รูปแบบคล้าย Linktree
แต่มีของเพิ่มที่ differentiates จากคู่แข่ง:
  - ลิงก์ BIO แบบพอร์ตโฟลิโอ (โปรไฟล์รวมลิงก์แพลตฟอร์ม)
  - แผนที่แสดง "คนใกล้เคียง" (Explore) ด้วยพิกัดจริง + ระยะ haversine
  - ระบบโหวตคำถามชุมชน (Vote)
  - ปรับธีมสีหน้าเว็บได้ "ต่อคน" (แต่ละโปรไฟล์มี ThemeConfig ของตัวเอง)
  - แผงแอดมิน + สถานะออนไลน์แบบ Realtime
  - ส่วน AI (ตัวการ์ตูน Mascot + แชท)

กลุ่มเป้าหมาย: ครีเอเตอร์/ฟรีแลนซ์/คนไทยทั่วไป ที่ต้องการหน้าพอร์ตโฟลิโอ
สั้นๆ มีสังคม + แผนที่ + โหวต

จุดแข็ง (USP): "BIO ลิงก์ + แผนที่คนใกล้เคียง + โหวต + ธีมต่อคน + AI"
ในตลาดไทยยังมีน้อยที่รวมหมดนี้ในที่เดียว

2. เทคโนโลยี (TECH STACK)
--------------------------------------------------------------------------------
หน้าบ้าน (Frontend):
  - React 19 + TypeScript
  - Vite 6 (build tool)
  - react-router-dom v7 — ใช้ "HashRouter" (URL เป็น /#/... เหมาะโฮสต์ static)

  - lucide-react (ไอคอน)
  - Leaflet (แผนที่ แทน Google Maps เพื่อไม่เสียค่า key)
  - PWA: manifest + icon + meta (ติดตั้งเป็นแอปได้ระดับเบื้องต้น)

หลังบ้าน / ฐานข้อมูล (Backend & DB):
  - Supabase = Postgres + Auth + Storage + Realtime
  - เรียกผ่าน @supabase/supabase-js (contexts/supabaseClient.ts)
  - โหมด Demo: ถ้าไม่มี .env แอปไม่พัง แต่สร้าง client แบบ fallback ทุก query มี
    try/catch + mock data → ดูหน้าเว็บได้โดยไม่ต่อ DB

AI:
  - Server/ai.js = serverless function เรียก Gemini 1.5 Flash
    (ใช้ env GEMINI_API_KEY ฝั่งเซิร์ฟเวอร์ — ถูกต้องด้านความปลอดภัย)
  - หมายเหตุ: README ระบุ AiMascot ใช้ Groq (VITE_AI_API_KEY ฝั่ง client)
    → มีความขัดแย้งระหว่างโค้ด (Gemini/server) กับเอกสาร (Groq/client)
    ควรเคลียร์ให้เหลือทางเดียวตอนทำเวอร์ชั่นสมบูรณ์

3. สถาปัตยกรรม (ARCHITECTURE)
--------------------------------------------------------------------------------
โครงสร้างโฟลเดอร์:
  pages/        → หน้าต่างๆ (Dashboard, Explore, Vote, Feed, Profile, Login, AdminPanel)
  components/   → Navbar, AiMascot, ErrorBoundary, GlassBackground, ThaiBackground,
                 Logo, DemoOverlay, ui/{Button, Card, Input, Modal}
  contexts/     → AuthContext.tsx (state กลางใหญ่), ThemeContext.tsx, supabaseClient.ts
  lib/          → designTokens.ts (ดีไซน์ซิสเต็ม), ranking.ts (จัดอันดับ+haversine),
                 social.ts (ตรวจจ่ายแพลตฟอร์มโซเชียล + ตรวจ URL ปลอดภัย)
  Server/       → ai.js (serverless AI)
  supabase/     → schema.sql + ไฟล์ migration หลายไฟล์ (idempotent)
  constants.ts  → จังหวัด 77 แห่ง + คำลอย (FLOATING_PHRASES)
  App.tsx       → Routing + Guard
  index.tsx     → Entry: ThemeProvider > AuthProvider > HashRouter > App
  index.html    → โหลดฟอนต์ IBM Plex + CSS variables + meta

Routing + Guard (App.tsx):
  - "/" และ "/login" → หน้า Login เสมอ (มีฟีดเบลอข้างหลัง)
  - "/feed" "/dashboard" "/explore" "/vote" "/admin" → ครอบด้วย <RequireAuth>
    ถ้าไม่ล็อกอินเด้งกลับ /login
  - "/:username" → หน้าโปรไฟล์ (เปิดสาธารณะ ไม่ต้องล็อกอิน)
  - ซ่อน Navbar/AiMascot/Popup บนหน้าล็อกอิน

State Management (AuthContext):
  เก็บ state กลางเยอะ: user, profile, usersList, posts, questions, popups,
  onlineUsers, allUsers, followingIds, aiConfigs, activePopup, isLoading
  + methods: login/register/logout/resetPassword, createPost/toggleLikePost/
  addComment/deletePost/uploadPostMedia, followUser/unfollowUser,
  loadAllUsers, banUser/unbanUser, createPopup/..., addQuestion/voteQuestion,
  askAiStylist, loadPosts, ฯลฯ
  จุดสังเกต: Context เดียวใหญ่มาก → เวอร์ชั่นสมบูรณ์อาจแตกเป็นหลาย context
  (Auth / Feed / Admin) หรือใช้ Zustand/Redux เพื่อลด re-render

Auth Flow:
  - สมัคร/ล็อกอินด้วยอีเมล+รหัสผ่าน (Supabase Auth)
  - "จำรหัสผ่าน" → เก็บ session ใน Supabase (localStorage 'zeelink_remember')
  - ตอนเปิดแอป getUser() กู้คืน session → loadUserByEmail → ถ้าโดนแบน signOut ทันที
  - onAuthStateChange → เลื่อนงาน async ด้วย setTimeout(0) เพื่อไม่ให้ deadlock
    กับ auth lock
  - Realtime Presence: channel 'zeelink-online' + heartbeat ทุก 30 วิ
    (ต้องรัน supabase/admin-online.sql ก่อนถึงจะเก็บ last_seen ลง DB)

6. ระบบจัดอันดับ (RANKING) — lib/ranking.ts
--------------------------------------------------------------------------------
Feed score (ยืมแนวคิด EdgeRank):
  feedScore = recency*0.5 + engagement*0.3 + affinity*0.2
  - recency: exponential decay ครึ่งชีวิต 48 ชม. (โพสต์ใหม่สำคัญสุด)
  - engagement: likes*1 + comments*2 + shares*1.5 แล้ว log10 เพื่อไม่ให้ viral โดมิเนต
  - affinity: ติดตามผู้โพสต์ +1.5
  เรียง O(n log n)

Explore ranking:
  ติดตามก่อน → จังหวัดเดียวกัน → ระยะใกล้ (haversine) → ยอดไลก์
  haversineKm(a,b): ระยะจริงกิโลเมตรระหว่าง 2 พิกัด (R=6371)

9. ช่องโหว่ / สิ่งที่ยังขาด (GAPS — สำคัญสำหรับเวอร์ชั่นสมบูรณ์)
--------------------------------------------------------------------------------
ด้านความปลอดภัย (HIGH PRIORITY):
  - RLS เปิดกว้างชั่วคราวใน schema.sql (เพราะยังไม่มี Auth แน่นหนา)
    → ก่อนเปิดสาธารณะ ต้องล็อก policy ให้เจ้าของดู/แก้ของตัวเองเท่านั้น
  - popup.content อาจใช้ dangerouslySetInnerHTML → เสี่ยง XSS ถ้า admin ใส่ HTML
    ไม่ปลอดภัย ควร sanitize หรือจำกัดเป็น text/markdown
  - AI key: README บอกใส่ VITE_AI_API_KEY (ฝั่ง client) → รั่วไหลได้
    โค้ด Server/ai.js ถูกแล้ว (key ฝั่งเซิร์ฟเวอร์) ต้องเคลียร์เอกสาร/การเรียกให้ตรง
  - Rate limit: ล็อกอิน/รีเซ็ต/โหวต ยังไม่มี rate limit ฝั่งแอป (พึ่ง Supabase บางส่วน)
  - Email confirmation: เปิด/ปิด auto-confirm ไม่ชัดเจน

ด้านฟีเจอร์ยังไม่เสร็จ:
  - AI "ต่อไฟล์จริง" ยังไม่เสร็จ (mascot config กับ serverless ยังไม่เชื่อมสมบูรณ์)
  - ยังไม่ deploy / ไม่มีระบบรายได้ (monetization)
  - ไม่มีระบบแจ้งเตือน (notifications) / แชทส่วนตัว
  - ไม่มี PWA offline / push notification จริง

ด้าน质量工程 (Quality Engineering):
  - ไม่มี automated tests (unit/integration) และไม่มี CI
  - ไม่มี error tracking (Sentry) / analytics
  - ไม่มี i18n (รองรับไทยอย่างเดียว)
  - HashRouter → SEO อ่อน (ถ้าต้องการ SEO ควร BrowserRouter + SSG/SSR)

ด้านประสิทธิภาพ (Performance):
  - Bundle ใหญ่ (~544 KB JS, 153 KB gzip) → ควร code-split/lazy load หน้า
    และแยก vendor (Leaflet โหลดเฉพาะตอนเข้าแผนที่)
  - รูป/มีเดียยังไม่มี optimization/CDN (พึ่ง picsum/ui-avatars ภายนอก)

ด้าน UX/Accessibility:
  - Mobile responsive มีหลายจุดที่อาจหลุด (ต้อง audit เพิ่ม)
  - ยังขาดการทดสอบผู้ใช้จริง / a11y audit
