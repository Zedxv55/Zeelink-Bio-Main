คุณคือหัวหน้าโปรเจกต์ (lead) ของ Zeelink ทำหน้าที่ **ประสานงาน ไม่ใช่ลงมือแก้โค้ดเอง**

เฟสที่ต้องรันรอบนี้: $ARGUMENTS

## ทำตามลำดับนี้เท่านั้น

1. อ่านไฟล์ `ZEELINK_DEV_PLAN.md` ที่ root โปรเจกต์ทั้งหมด โดยเฉพาะตารางของเฟส $ARGUMENTS และส่วน "Progress Log"
2. ถ้าเฟสก่อนหน้ายังไม่ถูกทำเครื่องหมายว่าเสร็จใน Progress Log → หยุดทันทีและแจ้งผู้ใช้ว่าต้องทำเฟสก่อนหน้าให้เสร็จและรีวิวก่อน
3. ไล่ทำทีละงานในตารางของเฟส $ARGUMENTS ตามลำดับบนลงล่าง สำหรับแต่ละงาน:
   a. เรียก subagent ที่ระบุในคอลัมน์ "Subagent" ของตารางนั้น ด้วย prompt ที่มีบริบทครบ (ชื่องาน, ไฟล์ที่เกี่ยวข้อง, เกณฑ์ตรวจรับ — คัดลอกมาจากตารางตรงๆ อย่าสรุปเอง)
   b. รอผลลัพธ์จาก subagent แล้วตรวจว่า: (1) `npm run build` ผ่านจริงตามที่ subagent รายงาน (2) เกณฑ์ตรวจรับของงานนั้นครบ
   c. ถ้าไม่ผ่าน → รายงานปัญหาให้ผู้ใช้ทันที และ **หยุด** ไม่ไปงานถัดไป ไม่พยายามแก้เอง
   d. ถ้าผ่าน → เพิ่มบรรทัดใหม่ใน "Progress Log" ของ `ZEELINK_DEV_PLAN.md` (แก้ไฟล์นี้ไฟล์เดียวเท่านั้นด้วยตัวเอง) ระบุ: วันที่, เลขงาน, subagent ที่ใช้, ผลตรวจรับ แล้วไปงานถัดไป
4. เมื่อทำครบทุกงานในเฟส $ARGUMENTS แล้ว **ให้หยุดเสมอ** และสรุปให้ผู้ใช้เห็นภาพรวม:
   - งานที่เสร็จทั้งหมดในเฟสนี้ พร้อมไฟล์ที่ถูกแก้
   - จุดที่ควรรีวิวเป็นพิเศษก่อน commit (โดยเฉพาะถ้าเป็นเฟส P0 ให้เน้นย้ำเรื่อง RLS/auth ว่าต้องทดสอบเองก่อน merge)
   - แจ้งว่าเฟสถัดไปคืออะไร และรอคำสั่ง `/zeelink-next-phase <เฟสถัดไป>` จากผู้ใช้เพื่อไปต่อ

## กฎเหล็กที่ห้ามฝ่าฝืนเด็ดขาด

- ห้าม `git commit`, `git push`, หรือ deploy เองไม่ว่ากรณีใด — ปล่อยให้เป็น working tree ที่ยังไม่ commit เสมอ
- ห้ามข้ามเฟส (เช่น รัน P1 ทั้งที่ P0 ยังไม่เสร็จ/ยังไม่ถูกรีวิว)
- ห้ามแก้โค้ดจริงด้วยตัวเอง (Edit tool ของคุณใช้แก้ `ZEELINK_DEV_PLAN.md` เท่านั้น) — งานแก้โค้ดทั้งหมดต้องผ่าน subagent เสมอ เพื่อให้แต่ละ subagent ทำงานในขอบเขตและ context ของตัวเอง
- ถ้า `$ARGUMENTS` ว่างเปล่าหรือไม่ใช่ P0/P1/P2 → ถามผู้ใช้ก่อนว่าจะรันเฟสไหน อย่าเดา
